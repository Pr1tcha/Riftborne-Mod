# Riftborne Rift Visual Context

Дата: 2026-06-13

## Общий контекст мода

`Riftborne Rift` - мод под Minecraft 1.21.1 NeoForge для сборки Riftborne. Лорно он связан с `Legion: Temporal Code` и сюжетным контекстом Riftborne. Основной текущий объект мода - разломы реальности.

Пользовательская роль в проекте:

- сюжет;
- квесты;
- общая сборка;
- интеграция мода в сборку;
- проверка визуала и геймплейных сценариев в игре.

Техническая роль Codex:

- код мода;
- визуальная система разломов;
- команды;
- рендер;
- интеграция библиотек;
- сборка и техническая проверка.

## Текущий технический стек

- Minecraft: 1.21.1
- Loader/API: NeoForge
- Java: JDK 21, локально использовался `C:\Program Files\Eclipse Adoptium\jdk-21.0.11.10-hotspot`

## Визуальное направление

Основной сюжетный разлом сейчас фиолетовый. Референс - образ Виталика: темная вертикальная трещина/портал, черно-фиолетовое ядро, светящиеся фиолетовые края, ощущение разрыва пространства.

Направление, которое понравилось:

- не обычная плоская текстура;
- процедурная форма;
- темное непрозрачное ядро;
- фиолетовые края;
- слабая дымка вокруг;
- пространственное искажение мира вокруг разлома;
- разлом должен выглядеть как физически опасная прореха в реальности.

План на будущее:

- у разных типов разломов будут разные цвета искажения;
- фиолетовый тип сейчас считается сюжетным;
- `Veil boom`/сильный взрыв реальности не должен быть постоянным эффектом вокруг стоящего разлома, его лучше использовать как отдельное событие активации/вспышки.

## История попыток с Veil distortion

Цель:

- сделать искажение пространства/мира вокруг разлома;
- эффект должен быть привязан строго к разлому;
- не должен жить отдельно от разлома;
- не должен уезжать по вертикали или горизонтали;
- должен пропадать на дистанции, когда сам разлом уже не читается.

### Самый удачный визуальный вариант

Самым удачным по ощущению был самый первый рабочий Veil distortion, примерно состояние со скрина `2026-06-13_12.46.21.png`.

Почему понравился:

- эффект был заметный;
- мир реально искажался;
- выглядело как разрыв пространства;
- визуально давал правильный масштаб и драму;
- вокруг разлома появлялась сильная screen-space деформация.

После этого попытки улучшать цвет, дальность и привязку начали ломать позиционирование.

### Что пробовали

1. Сделать distortion фиолетовым вместо радужного/cyan-violet.
2. Добавить distance falloff.
3. Добавить жесткий cutoff по расстоянию.
4. Считать экранные координаты через матрицу рендера procedural billboard.
5. Считать bounds разлома по фактическим точкам сегментов.
6. Отключить distortion для `SCAR`.
7. Выбирать лучший разлом в кадре по близости к центру экрана.

### Что сработало

- Distance cutoff/falloff сработал хорошо: на дальности distortion перестает жить отдельно.
- Сам shader первого distortion визуально хороший и его стоит сохранить как базовую точку.
- Фиолетовая дымка/обводка самого procedural-разлома выглядит нормально.

### Что не сработало

Главная проблема: привязка distortion к разлому.

Текущее наблюдение пользователя:

- если прицел/центр экрана наведен на разлом, distortion становится как положено;
- если смотреть не строго на разлом, distortion уезжает;
- уезд особенно заметен по вертикали, иногда немного по горизонтали;
- эффект может появляться рядом, ниже или в стороне от разлома;
- это неприемлемо: distortion должен исходить только от самого разлома.

Отдельная проблема:

- в кадре могут быть остатки/другие rift-объекты;
- post-process uniform один на весь экран;
- если несколько кандидатов пишут параметры в один pipeline, эффект может выбирать не тот источник;
- была попытка запретить `SCAR` писать distortion и выбирать лучший живой candidate, но это не решило фундаментальную проблему.

## Точное описание текущего бага

Veil distortion сейчас является screen-space post-process эффектом, который получает uniform-параметры:

- центр эффекта на экране;
- радиус эффекта;
- сила эффекта;
- время анимации.

Проблема в том, что центр эффекта вычисляется отдельно от того, как Minecraft/renderer фактически выводит procedural-разлом на экран. Поэтому при изменении угла камеры экранная позиция эффекта и экранная позиция самого разлома расходятся.

Симптомы:

- distortion корректно совпадает с разломом только когда разлом близко к центру экрана;
- при смещении камеры distortion отъезжает;
- чем дальше разлом от центра взгляда, тем заметнее рассинхрон;
- вертикальный уезд сильнее горизонтального;
- если в кадре есть другие rift-объекты, эффект может выглядеть так, будто относится не к активному разлому.

Требование:

- distortion должен быть жестко приклеен к экранному силуэту разлома;
- если разлом сдвинулся на экране, distortion обязан сдвинуться ровно вместе с ним;
- если разлом не рендерится или слишком далеко, distortion не должен рендериться;
- цвет потом можно менять, но сначала нужно восстановить стабильную привязку.

## Вероятный технический вывод

Эллипсный screen-space эффект с ручным вычислением центра слишком хрупкий. Он может быть приемлем только как временный вариант.

Для настоящей жесткой привязки лучше перейти к mask-based подходу:

1. Отрисовать маску разлома в отдельный буфер/слой.
2. Veil post-process должен читать эту маску.
3. Искажение применяется только там, где есть маска разлома или расширенная область вокруг маски.
4. Центр/радиус не угадываются вручную.
5. Несколько разломов в кадре работают естественно, потому что каждый пишет в маску.

Это должно решить:

- уезд по вертикали;
- уезд по горизонтали;
- конфликт нескольких разломов;
- эффект без тела разлома;
- необходимость угадывать screen position.

## Рекомендованный следующий шаг

Не продолжать крутить текущую формулу центра.

Следующий технический шаг:

- сохранить первый удачный shader distortion как визуальную базу;
- попробовать сделать отдельный Veil mask/pass для procedural-разлома;
- distortion применять не по uniform-эллипсу, а по маске;
- после стабильной привязки уже менять цвет с cyan-violet на фиолетовый.
